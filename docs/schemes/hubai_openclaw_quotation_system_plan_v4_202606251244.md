# OpenClaw 统一入口报价系统：规划与部署方案

> 版本：v4.1
> 时间：2026-06-25
> 定位：以 OpenClaw Gateway 为接入中枢，IM 层覆盖主流企业应用，接入层统一 OpenClaw/Agents/Web 三类入口，对接 HubAI 报价工作流 v3.0 底座
> 核心原则：入口统一、渠道无差别、规则外置、历史隔离、Agent 协同
> 架构优化：原「交互层」拆分为「IM 层」+「接入层」两层，分层更清晰

---

## 一、项目背景与目标

### 1.1 项目背景

- 现有报价入口分散：Web 页面、各 IM 平台人工流转、Agent 零散调用
- 报价规则不统一：不同渠道引用历史项目方式不一致，审批流程不可追溯
- 多 Agent 能力未协同：商务、技术、财务 Agent 各自为政，缺乏统一分流与协同机制
- 信息孤岛：客户需求、产品数据、历史案例、审批记录分散在不同系统

### 1.2 核心目标

| 目标 | 说明 |
|---|---|
| **IM 层全覆盖** | 接入 WebChat、钉钉、飞书、企微及主流 OA 等企业应用 |
| **接入层统一** | OpenClaw、其他 Agents 智能体、Web 管理后台三类接入方式共用一套业务 API |
| **流程闭环** | 需求录入 → 规则校验 → 缺口发现 → 分流派单 → 方案生成 → 报价审批 → 客户输出 → 知识库沉淀 |
| **规则可控** | 历史项目仅作参考，报价规则（价格、折扣、审批）集中配置、可追溯、可审计 |
| **多 Agent 协同** | 商务、技术、财务 Agent 按分流规则自动介入，子 Agent 通过 `sessions_spawn` 完成专项任务 |
| **快速落地** | 复用现有 HubAI 报价底座（v3.0），MVP 先行、分阶段迭代 |

## 二、总体架构设计

### 2.1 分层架构

```text
┌─────────────────────────────────────────────────────────────────────┐
│                           IM 层 (企业应用入口)                        │
│  ┌──────────┬──────────┬──────────┬──────────┬──────────────────┐  │
│  │ WebChat  │   钉钉   │   飞书   │   企微   │  OA / 其他系统   │  │
│  └──────────┴──────────┴──────────┴──────────┴──────────────────┘  │
├─────────────────────────────────────────────────────────────────────┤
│                         接入层 (三类入口统一)                        │
│  ┌────────────────────┬────────────────────┬─────────────────────┐ │
│  │  OpenClaw Gateway  │  其他 Agents 智能体  │   Web 管理后台      │ │
│  │  ├─ 通道插件        │  ├─ sessions_spawn   │  ├─ 报价工作台      │ │
│  │  ├─ bindings 路由   │  ├─ bindings 绑定     │  ├─ 需求表单        │ │
│  │  └─ Hook 入口       │  └─ 结果回流          │  └─ 审批看板        │ │
│  └────────────────────┴────────────────────┴─────────────────────┘ │
├─────────────────────────────────────────────────────────────────────┤
│                    报价工作流引擎 (HubAI Quotation v3)               │
│  ┌──────────┬──────────┬──────────┬──────────┬──────────┬──────────┐│
│  │ 需求表三 │ 规则校验 │ 缺口发现 │ 分流派单 │ 方案生成 │ 报价草稿 ││
│  ├──────────┼──────────┼──────────┼──────────┼──────────┼──────────┤│
│  │ 规则校验 │ 审批流   │ 客户版本 │ 知识沉淀 │ 状态同步 │ 日志审计 ││
│  └──────────┴──────────┴──────────┴──────────┴──────────┴──────────┘│
├─────────────────────────────────────────────────────────────────────┤
│                           数据与知识层                                │
│  ┌──────────┬──────────┬──────────┬──────────────────────────────┐  │
│  │ 产品库   │ 价格政策 │ 历史项目 │        IMA 知识库             │  │
│  │ 客户档案 │ 库存数据 │ 合同模板 │ 需求库/报价库/方案库/过程档案 │  │
│  └──────────┴──────────┴──────────┴──────────────────────────────┘  │
└─────────────────────────────────────────────────────────────────────┘
```

### 2.2 核心架构原则

1. **入口统一**：所有 IM 渠道和接入方式最终落地同一套 `order_requirement` 需求表三与报价流程
2. **渠道无差别**：WebChat、钉钉、飞书、企微、OA 仅为交互入口，不承载业务规则
3. **接入方式无差别**：OpenClaw、其他 Agents、Web 后台三类接入方式共用同一套业务 API
4. **协议统一**：无论 OpenClaw 还是其他 Agents，都通过同一套 MCP 协议调用报价能力
5. **规则引擎外置**：报价政策、折扣、审批规则集中配置，不硬编码在渠道插件或接入层
6. **历史项目隔离**：历史项目仅用于相似性检索与参考，不直接参与定价计算
7. **会话隔离**：不同渠道、不同客户、不同 Agent 的会话按 OpenClaw session key 隔离

### 2.3 MCP 协议适配层（薄接入层）

在接入层和报价工作流引擎之间增加一层 MCP 协议适配，**不推翻原有 HubAI API 设计**：

```text
┌─────────────────────────────────────────────────────────────────────┐
│                           接入层                                      │
│  ┌──────────────────┬──────────────────┬──────────────────┐        │
│  │   OpenClaw       │  其他 Agents     │   Web 管理后台    │        │
│  └──────────────────┴──────────────────┴──────────────────┘        │
├─────────────────────────────────────────────────────────────────────┤
│                      MCP 协议适配层（薄接入层）                       │
│  ┌───────────────────────────────────────────────────────────┐    │
│  │  MCP Server：封装 HubAI 报价 API，暴露标准协议接口         │    │
│  │  ├─ create_requirement()       # 创建需求表三              │    │
│  │  ├─ generate_requirement_card() # 生成需求卡                │    │
│  │  ├─ route_requirement()        # 需求分流                  │    │
│  │  ├─ get_historical_cases()     # 获取历史案例              │    │
│  │  ├─ generate_quotation()       # 生成报价草稿              │    │
│  │  ├─ validate_quotation()       # 规则校验                  │    │
│  │  ├─ approve_quotation()        # 审批操作                  │    │
│  │  └─ generate_customer_output() # 生成客户版本              │    │
│  └───────────────────────────────────────────────────────────┘    │
├─────────────────────────────────────────────────────────────────────┤
│                    HubAI 报价系统 API（完全保留）                     │
└─────────────────────────────────────────────────────────────────────┘
```

**MCP 适配层的价值**：

| 价值 | 说明 |
|---|---|
| **协议统一** | 所有 Agent 用同一套 MCP 协议调用，不需要每个 Agent 适配 HubAI API 细节 |
| **能力标准化** | 报价能力成为标准 MCP 工具，可跨 Agent、跨平台复用 |
| **向后兼容** | 原有 HubAI API 100% 保留，MCP 层只是做协议转换，不修改业务逻辑 |
| **渐进式落地** | MVP 阶段可先做 3-5 个核心 MCP 方法，后续逐步补充完整 |
| **不推翻架构** | 只是在接入层和业务层之间加一层薄适配，架构主体不变 |

## 三、渠道接入方案

### 3.1 IM 层接入矩阵

| IM 渠道 | 接入层归属 | 接入方式 | 当前状态 | 支持能力 | 优先级 |
|---|---|---|---|---|---|
| **WebChat** | OpenClaw | OpenClaw 内置 | ✅ 已可用 | 全功能 | P0 |
| **钉钉** | OpenClaw | 现有 `dingtalk-openclaw-bridge` | ✅ 已部署 | 消息、文件、审批回调 | P0 |
| **飞书** | OpenClaw | OpenClaw 内置 Feishu 通道 | ⚠️ 待配置 | 消息、文件、群组、卡片 | P0 |
| **企微** | OpenClaw | Webhook + 企业微信插件适配 | ⚠️ 待接入 | 消息、文件、群组 | P1 |
| **OA/其他系统** | Web | API 对接 + Webhook | ⚠️ 待规划 | 审批回调、需求同步 | P2 |
| **Web 管理后台** | Web | 前端页面 + 统一业务 API | ⚠️ 待开发 | 工作台、表单、审批、统计 | P1 |

### 3.2 接入层分类说明

| 接入方式 | 覆盖场景 | 核心能力 |
|---|---|---|
| **OpenClaw** | WebChat + IM 渠道接入 | 通道插件、会话路由、Hook 入口、多 Agent 调度 |
| **其他 Agents 智能体** | 外部 Agent 调用、子任务分发 | `sessions_spawn`、bindings 绑定、结果回流 |
| **Web 管理后台** | 人工操作、批量处理、审批看板 | 报价工作台、需求表单、审批管理、统计报表 |

### 3.3 钉钉接入方案（P0）

- **复用现有连接器**：`dingtalk-openclaw-bridge` 作为 OpenClaw 通道插件运行
- **消息路由**：群聊 @机器人 / 单聊 → OpenClaw 事件 → 报价 Skill 触发
- **审批对接**：钉钉审批流 Webhook → OpenClaw Hook → 更新报价单状态
- **身份映射**：钉钉 `userid` ↔ OpenClaw `allowFrom` 白名单 ↔ 报价系统 `sales_owner_user_id`
- **入口字段**：`source_channel = "dingtalk"`，`source_ref = 钉钉消息ID`

### 3.4 飞书接入方案（P0）

- **内置通道启用**：OpenClaw 2026.5.29+ 原生支持 Feishu
- **配置步骤**：
  1. `openclaw channels login --channel feishu`
  2. 粘贴飞书开放平台 App ID / App Secret
  3. `openclaw gateway restart`
- **群组策略**：`groupPolicy = allowlist`，指定报价群 ID；默认 `requireMention = true`
- **DM 策略**：`dmPolicy = pairing`，商务团队成员白名单放行
- **入口字段**：`source_channel = "feishu"`，`source_ref = 飞书 open_chat_id`

### 3.5 企微接入方案（P1）

- **阶段一（MVP）**：企微群机器人 Webhook → OpenClaw `/hooks/agent` 入口
- **阶段二**：企业微信渠道插件接入，支持双向消息、文件、上下文
- **路由方式**：Webhook payload 携带 `source_channel = "wecom"`，统一路由到报价 Skill
- **配置示例**：
  ```json5
  {
    hooks: {
      enabled: true,
      token: "shared-secret",
      path: "/hooks",
    }
  }
  ```

### 3.6 其他 Agents 智能体接入方案（P1）

### 3.7 Web 管理后台接入方案（P1）

- **定位**：人工操作入口、批量处理、审批看板、统计报表
- **功能清单**：
  - 报价工作台：查看所有待处理报价、需求表三列表
  - 需求表单：Web 端可视化录入客户需求、上传附件
  - 审批管理：审批人查看待审批项、一键通过/驳回/批注
  - 统计报表：报价数量、转化率、平均周期、产品热销排行
  - 系统配置：价格政策、折扣规则、审批流配置、渠道白名单
- **技术实现**：
  - 前端：Vue.js / React，复用现有 HubAI Web 端框架
  - 后端 API：与 OpenClaw 调用同一套 HubAI 报价 API
  - 认证：企业 SSO（钉钉/飞书/企微扫码登录）
  - 权限：RBAC 角色权限控制（管理员/商务/技术/财务/审批人）
- **入口字段**：`source_channel = "web"`，`source_ref = "web-admin"`

### 3.8 OA 及其他企业应用接入方案（P2）

- **通用对接模式**：OA 系统 → Webhook → OpenClaw `/hooks/agent` → 报价流程
- **典型场景**：
  - OA 提交「报价申请单」→ 自动创建需求表三
  - OA 审批通过 → 自动触发报价生成
  - 报价结果回写到 OA 附件或审批意见
- **接入原则**：OA 系统只做入口和通知，不承载报价业务规则

- **Agent 定义**：OpenClaw `agents.list` 统一管理主 Agent 和其他 Agents 智能体
  ```json5
  {
    agents: {
      list: [
        { id: "main", name: "主助理", workspace: "~/.openclaw/workspace" },
        { id: "commerce", name: "商务助理", workspace: "~/.openclaw/workspace-commerce" },
        { id: "tech", name: "技术助理", workspace: "~/.openclaw/workspace-tech" },
        { id: "finance", name: "财务助理", workspace: "~/.openclaw/workspace-finance" }
      ]
    }
  }
  ```
- **路由绑定**：`bindings` 按需求类型分配 Agent
- **子任务编排**：`sessions_spawn` 启动子 Agent 完成专项任务
- **结果回流**：子 Agent 结果通过 `message:sent` Hook 回填主报价会话

## 四、报价工作流核心模块规划

### 4.1 需求接入层（客户需求表三）

所有入口强制生成需求表三记录，字段包括：

| 字段类别 | 具体内容 |
|---|---|
| 客户信息 | 客户名称、联系人、电话/邮箱、客户等级 |
| 需求主体 | 产品清单（产品编码、数量）、定制化需求描述 |
| 交付与条款 | 预算范围、期望交付时间、付款方式、账期要求 |
| 渠道信息 | `source_channel`、`source_ref`、`created_by` |
| 历史项目 | `has_historical_project`、历史项目引用链接 |
| 输出选择 | 是否需要方案、是否需要报价单 |

- **前端交互**：Web 端表单 / IM 端多轮追问 + AI 结构化提取
- **强约束**：未保存需求表三，任何渠道不得进入正式报价流程

### 4.2 规则校验与缺口发现

| 校验项 | 规则 | 缺失处理 |
|---|---|---|
| 完整性 | 客户名称、产品、数量、预算、交付期必填 | 自动生成追问清单 |
| 产品匹配 | `product` 表中是否存在匹配记录 | 标记 blocker，转技术助理 |
| 价格匹配 | `product_price` 是否存在 active 记录 | 标记 high，商务确认 |
| 库存匹配 | `inventory.available >= 需求数量` | 标记 high，确认分批发货 |
| 政策匹配 | `quotation_policy` 是否覆盖客户等级+品类 | 标记 medium，走审批 |
| 账期合规 | 是否超过标准政策 | 标记 medium，走财务审批 |
| 风险标记 | 急单/低毛利/新客户大额 | 标记 risky，升级审批 |

### 4.3 需求分流引擎

| 需求类型 | 判定条件 | 负责 Agent | 审批要求 | 预计时间 |
|---|---|---|---|---|
| **标准型** | 产品在库、价格政策匹配、库存充足、账期合规 | 商务助理（自动） | 无需人工审批 | 5 分钟 |
| **方案型** | 多产品组合、需技术方案说明 | 商务 + 技术助理 | 技术方案确认 | 1-2 天 |
| **非标型** | 含定制需求、服务交付内容 | 商务 + 技术 + 财务 | 三方评审 | 3-5 天 |
| **高风险** | 账期超长、毛利过低、新客户大额 | 商务总监 + 财务 | 负责人/老板审批 | 待定 |

**分流决策树**：
```text
是否有 risk_flags？
  ├── 是 → 高风险型 → 升级审批
  └── 否 → 是否有 custom_requirements？
         ├── 是 → 非标型 → 内部评审
         └── 否 → 产品是否明确？
                ├── 否 → 方案型 → 技术确认
                └── 是 → 标准型 → 快速报价
```

### 4.4 历史项目参考引擎

- **检索范围**：历史报价单、历史合同、已交付项目知识库
- **匹配维度**：产品相同(40%) + 数量相近(20%) + 客户级别(15%) + 交付方式(10%) + 时间(10%) + 行业(5%)
- **输出内容**：
  - Top 3 相似项目（相似度、产品组合、金额、折扣、交付周期、赢输单结果）
  - 风险提示（历史交付逾期、客户投诉点、变更记录）
- **关键约束**：页面和输出文案必须标注 **"仅作参考，不作为报价规则"**
- **技术实现**：IMA 知识库检索 + 关键词匹配（暂不引入向量计算）

### 4.5 报价生成与规则引擎

**价格来源优先级**：
1. `product_price` 最新 active 价格
2. `quotation_policy` 最优折扣（客户级别 × 数量 × 品类）
3. `inventory` 库存校验
4. 历史案例只写入 `historical_case_refs`，不参与定价

**约束检查**：
- 折扣下限锁定（低于下限自动触发审批）
- 库存预留校验
- 账期与付款方式合规校验

**输出**：
- 报价单草稿（Markdown / PDF）
- 方案文档（方案型需求）
- 审批流触发条件

### 4.6 审批与客户输出

- **审批路径**：按分流结果自动路由，IM 端推送审批卡片
- **客户输出模板**：
  - Markdown 报价单（适配所有 IM 渠道）
  - PDF 附件（Web 端下载）
  - 方案说明文档（方案型需求）
- **状态同步**：审批结果、客户反馈实时同步回需求表三

### 4.7 知识库沉淀

报价完成后自动归档到 IMA 知识库：

| 归档内容 | 目标知识库 | 标签 |
|---|---|---|
| 需求表三 | 报价需求库 | 客户、产品、时间 |
| 最终报价单 | 历史报价库 | 产品、金额、赢输单 |
| 方案文档 | 解决方案库 | 行业、产品组合 |
| 审批意见与变更 | 过程档案 | 需求编号、审批人 |

## 五、Skill 与能力封装

### 5.1 报价核心 Skill (`quotation-core`)

- **触发词**：「报价」「做个报价」「给客户报个价」「查报价」
- **能力矩阵**：

| 能力 | 说明 |
|---|---|
| 多轮追问 | 补全需求表三必填字段 |
| 规则校验 | 完整性评分、缺口发现 |
| 分流判定 | 标准/方案/非标/高风险识别 |
| Agent 指派 | 按分流结果路由到对应 Agent |
| 报价草稿 | 基于价格政策生成报价单 |
| 状态查询 | 「我的报价」「待审批报价」「历史报价」 |

### 5.2 历史项目参考 Skill (`quotation-history`)

- **触发**：需求表三保存后自动执行，或用户主动「查历史项目」「类似案例」
- **输入**：客户名称、产品清单、行业、预算范围
- **输出**：相似案例列表 + 风险提示 + "仅供参考" 标注

### 5.3 渠道适配层 Skill

| Skill | 职责 |
|---|---|
| `dingtalk-adapter` | 解析钉钉消息格式、@提醒、审批回调、附件下载 |
| `feishu-adapter` | 解析飞书富文本消息、卡片消息、群成员身份 |

### 6.2 部署清单

| 组件 | 部署方式 | 资源要求 | 备注 |

### 6.3 网络与安全

| 层面 | 措施 |
|---|---|
| **通道回调** | HTTPS + IP 白名单（钉钉/飞书/企微服务器网段） |
| **Hook 接口** | `Authorization: Bearer <token>` 鉴权，query-string token 拒绝 |
| **文件权限** | 工作区与数据库文件 `0600` 权限 |
| **备份策略** | 每日自动备份 SQLite + 工作区文件到异地 |
| **渠道白名单** | `allowFrom` 限制可发起报价的用户范围 |
| **会话隔离** | Session key 按 `agent:<agentId>:<channel>:<peer>` 隔离 |

## 八、风险控制与验收标准

### 8.1 风险清单与应对

| 风险 | 影响 | 概率 | 应对措施 |
|---|---|---|---|
| 钉钉/飞书 API 变更导致消息中断 | 高 | 中 | 保留 Webhook 降级方案；定期监控通道状态 |
| 规则引擎误判导致报价错误 | 高 | 低 | 报价草稿强制人工确认；新增规则灰度发布 |
| IMA 知识库检索超时 | 中 | 中 | 增加本地关键词检索作为降级；结果缓存 5 分钟 |
| 多 Agent 并发导致会话混乱 | 中 | 低 | Session key 按渠道+客户+需求ID隔离；超时自动归档 |
| 企微插件适配周期超预期 | 中 | 高 | MVP 先用 Webhook 方案；插件适配延后到 M4 |
| 报价数据量增长导致 SQLite 性能下降 | 低 | 中 | M4 后评估迁移 PostgreSQL；当前先加索引优化 |

### 8.2 验收标准

| 验收项 | 标准 |
|---|---|
| **入口覆盖** | WebChat、钉钉、飞书、企微至少一个可完成完整报价流程 |
| **需求表三** | 所有入口提交的数据落到同一 `order_requirement` 表，`source_channel` 正确 |
| **规则校验** | 必填字段缺失时阻断报价，自动生成追问清单 |
| **分流准确** | 标准/方案/非标/高风险四类需求识别准确率 ≥ 90% |
| **历史隔离** | 历史项目展示在报价单中标注"仅供参考"，不参与价格计算 |
| **审批闭环** | 非标/高风险需求必须经审批后才可输出客户版本 |
| **知识沉淀** | 报价完成后 30 秒内可在 IMA 知识库检索到归档记录 |
| **多 Agent** | 技术/财务 Agent 可按分流规则自动介入并返回结果 |
