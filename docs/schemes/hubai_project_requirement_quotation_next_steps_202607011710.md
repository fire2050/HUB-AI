# HubAI 项目需求报价设计下一步建议

> 时间：2026-07-01 17:10  
> 背景：单品快速查价基本完成，下一阶段转入“项目需求报价 / 方案报价 / 客户正式报价”能力设计。  
> 基准方案：`hubai_unified_solution_v1.3_202606291723.md`、`hubai_quotation_workflow_design_v3_202606250952.md`、当前 `smart-quotation` 与 `product-wuying-pc` Skill。

---

## 一、当前状态判断

单品报价已经解决的是“产品价格查询”问题，但项目需求报价要解决的是“客户正式报价流程”问题，二者边界必须继续保持清晰：

| 类型 | 当前状态 | 下一步处理 |
|---|---|---|
| 单品查询 | 基本完成 | 作为内部参考报价能力保留 |
| 项目需求报价 | 进入设计阶段 | 必须走客户需求表三 |
| 方案报价 | 尚未完整闭环 | 需要需求卡、分流、方案草稿、报价草稿、审批 |
| 客户正式报价单 | 尚未完整闭环 | 必须经过规则校验和人工确认 |

项目需求报价的核心不是“多查几个产品价格”，而是要把客户需求沉淀为结构化项目、再驱动报价流程。

---

## 二、必须继承的已有约束

根据历史方案和当前规则，项目需求报价必须保留以下约束：

1. **客户需求表三必经**  
   任何客户/项目/方案/P0/正式报价，不得直接输出正式报价，必须先创建或补齐客户需求表三。

2. **历史项目仅参考**  
   历史案例、历史报价可以用于辅助判断风险、方案表达和销售建议，但不能替代当前产品价格表、政策、库存和审批规则。

3. **认知与执行分离**  
   大模型负责理解、提取、追问、调度；Skill 负责确定性计算、规则校验、文档生成。

4. **多渠道统一入口**  
   Web、钉钉、企微、飞书、其他 Agent 入口都必须进入同一套项目报价 API / Skill 流程。

5. **正式报价需人工确认**  
   对客户输出前必须经过规则校验、报价草稿确认、审批或人工确认。

## 三、建议的项目需求报价主流程

建议把项目需求报价定义为 8 个阶段：

```text
入口识别
  ↓
客户需求表三
  ↓
需求卡生成
  ↓
缺口追问
  ↓
分流派单
  ↓
方案/报价草稿
  ↓
规则校验 + 人工确认
  ↓
客户版本输出 + 回写知识库
```

### Step 1：入口识别

继续复用当前统一入口分类器：

| flow_type | 处理 |
|---|---|
| `single_item` | 单品快速查价 |
| `requirement_p0` | 项目需求报价 / 方案报价 / 客户正式报价 |
| `unclear` | 先追问入口类型 |

项目报价触发词包括：客户、项目、方案、正式报价、客户报价单、P0、批量采购、部署、交付、预算、招投标、合同、客户需求等。

### Step 2：客户需求表三

项目需求报价必须先生成客户需求表三。建议字段分为 5 类：

| 类别 | 必填字段 | 说明 |
|---|---|---|
| 客户信息 | 客户公司、联系人、联系方式 | 项目归属和后续跟进 |
| 项目信息 | 项目名称、需求来源、客户原文 | 保留原始上下文 |
| 产品需求 | 产品线、配置、数量、时长类型、硬件设备 | 进入报价计算的基础 |
| 交付需求 | 使用场景、交付时间、部署方式、培训/安全要求 | 判断方案复杂度 |
| 商务需求 | 预算、付款方式、竞品、特殊条款 | 判断风险和审批 |

无影产品建议补充一个专属必填字段：**时长类型**。

可选值：

| 序号 | 时长类型 |
|---:|---|
| 1 | 120小时 |
| 2 | 200小时 |
| 3 | 不限时长（1小时休眠） |
| 4 | 不限时长 |
| 5 | 教育办公 |

### Step 3：需求卡生成

需求卡不是报价单，而是报价前的“项目体检单”。建议输出：

| 模块 | 内容 |
|---|---|
| 完整性评分 | 0-100 分 |
| 缺失字段 | 哪些必填信息还没拿到 |
| 风险标签 | 超短交期、预算不足、账期风险、非标交付等 |
| 产品匹配 | 推荐产品/配置/数量是否可识别 |
| 追问清单 | 下一轮需要问客户或销售的问题 |
| 分流建议 | 标准型 / 方案型 / 非标型 / 高风险 |

### Step 4：缺口追问

追问要按优先级，不要一次问太多。建议分三层：

| 优先级 | 字段 | 处理 |
|---|---|---|
| P0 必问 | 客户公司、产品/配置、数量、时长类型、交付时间 | 不完整则不能报价 |
| P1 强建议 | 使用场景、预算、联系方式、交付方式 | 缺失可生成待确认草稿 |
| P2 可后补 | 竞品、培训、安全、账期、历史项目 | 用于方案优化和风险判断 |

钉钉对话建议每次最多追问 3-5 个字段，避免消息过长。

### Step 5：分流派单

项目需求建议分四类：

| 类型 | 判断条件 | 输出 |
|---|---|---|
| 标准型 | 单产品/少量配置、字段完整、无特殊交付 | 报价草稿 |
| 方案型 | 多产品组合、需要技术说明、客户关注场景价值 | 方案草稿 + 报价草稿 |
| 非标型 | 定制、集成、复杂部署、特殊安全要求 | 派技术/商务协同 |
| 高风险 | 账期长、低毛利、超短交期、大额新客户、越权折扣 | 必须审批 |

现有 `route_engine.py` 已有四类路由雏形，下一步应增强规则来源和输出解释。

### Step 6：方案/报价草稿

根据输出选择，生成三种内部草稿：

| 输出类型 | 内容 |
|---|---|
| 仅报价 | 产品明细、数量、单价、合计、有效期、备注 |
| 仅方案 | 客户需求摘要、产品组合、部署建议、交付计划 |
| 方案 + 报价 | 方案说明 + 标准报价表 |

注意：此阶段仍是**内部草稿**，不能直接发客户。

### Step 7：规则校验 + 人工确认

正式客户报价前必须校验：

| 检查项 | 说明 |
|---|---|
| 产品存在 | 产品必须来自当前有效产品库 |
| 价格完整 | 所有报价项必须有当前价格 |
| 时长一致 | 无影产品同一报价中不得混用未确认时长 |
| 数量有效 | 数量必须为正数 |
| 库存/交付 | 如有库存或交付限制必须提示 |
| 折扣权限 | 折扣或特殊价必须走审批 |
| 风险标签 | 高风险报价必须拦截客户输出 |

### Step 8：客户版本输出 + 回写知识库

客户版本必须脱敏，只保留客户可见信息：

- 客户公司
- 报价日期 / 有效期
- 产品明细
- 数量
- 月价 / 年价 / 合计
- 交付说明
- 商务条款
- 联系方式

内部信息如毛利、成本、审批意见、历史项目对比，不进入客户版。

## 四、无影产品项目报价的字段建议

相比单品查询，项目报价至少需要这些字段：

| 字段 | 是否必填 | 说明 |
|---|---|---|
| 报价类型 | 是 | 固定为方案报价/项目报价 |
| 客户公司 | 是 | 客户需求表三基础字段 |
| 联系人/联系方式 | 是 | 后续交付和跟进 |
| 使用场景 | 是 | 安全办公、教育培训、研发设计、数字人&媒体运营、AI agents、其他 |
| 部署规模 | 是 | 点位数/用户数 |
| 云产品资源配置 | 是 | 必须来自产品库 |
| 时长类型 | 是 | 120小时/200小时/不限时长等 |
| 休眠方式 | 是 | 支持24小时开机/无强制休眠策略 |
| 硬件设备 | 否 | 可多选，可不需要 |
| 云盘需求 | 否 | 不明确则不估算 |
| 数据安全需求 | 否 | 用于方案表达 |
| 预算范围 | 建议 | 用于风险判断，不作为定价规则 |
| 交付时间 | 建议 | 判断超短交期风险 |
| 付款方式 | 建议 | 判断账期风险 |
| 竞品信息 | 可选 | 用于销售建议 |

## 五、下一步实施建议

建议按 4 个小步推进，不要一次做大闭环。

### M1：项目需求表三最小闭环

目标：让 Web/钉钉可以创建项目需求，不直接报价。

要做：

1. 统一项目需求字段 Schema。
2. 完善 `product-wuying-pc/config/requirement.json`，补充 `duration_type`、`project_name`、`payment_terms`、`output_type` 等字段。
3. 创建/修正 `order_requirement_table` 与明细表字段。
4. 增加 API：
   - `POST /api/quotation/requirements`
   - `GET /api/quotation/requirements/{entry_id}`
5. 钉钉命中项目报价时，回复“已进入客户需求表三流程”，并追问 P0 字段。

验收：用户说“给客户做无影项目报价”，系统不能直接报价，而是生成需求表三草稿并追问缺失字段。

### M2：需求卡 + 缺口追问

目标：让系统知道“还能不能报价”。

要做：

1. 重构 `requirement_card.py`。
2. 完整性评分从硬编码产品/公司扩展到客户、产品、数量、时长、交付、预算等字段。
3. 自动生成追问清单。
4. 钉钉追问格式标准化，每次最多 3-5 个问题。

验收：输入一段客户原文，系统能输出需求卡、缺失字段、下一步追问。

### M3：项目报价草稿

目标：字段完整后生成内部报价草稿。

要做：

1. 项目报价引擎聚合：云产品资源 + 硬件设备 + 云盘/带宽等可选项。
2. 统一报价表格式。
3. 明确月价、年价、数量、小计、合计。
4. 生成内部版 Markdown 报价草稿。

验收：一个标准型无影项目，能生成内部报价草稿，但标记为“待人工确认”。

### M4：审批与客户版输出

目标：从内部草稿走向客户报价单。

要做：

1. 增加审批状态机：draft → validated → pending_approval → approved → customer_ready。
2. 生成客户版 Markdown/Excel。
3. Web 支持下载客户版报价单。
4. 报价结果回写知识库。

验收：客户版输出不含内部成本、毛利、审批意见；可下载、可归档。

## 六、优先级建议

我建议下一步先做 **M1 + M2**，不要立即做完整报价单生成。

原因：

1. 项目报价的最大风险不是算价，而是需求不完整。
2. 客户需求表三和需求卡是后续报价、审批、客户版输出的共同底座。
3. 单品报价已经证明价格数据可用，现在需要证明“项目需求能被结构化管理”。
4. 先把“不能绕过需求表三”落实到 Web 和钉钉，后续才不会越做越乱。

建议立刻开始的开发任务：

| 优先级 | 任务 | 产物 |
|---|---|---|
| P0 | 定义项目需求表三字段 v1 | JSON Schema + DB 表 |
| P0 | 无影项目报价 P0 必填字段 | requirement.json 更新 |
| P0 | 需求卡完整性评分 | requirement_card.py |
| P0 | 钉钉项目报价入口追问 | dingtalk-connector / smart-quotation |
| P1 | Web 需求表三页面 | 表单 + 详情页 |
| P1 | 内部报价草稿 | Markdown 表格 |
| P2 | 客户版 Excel/PDF | 下载与归档 |

## 七、建议的标准对话体验

### 用户：

```text
给 XX 公司做一个无影云电脑项目报价，50 人，安全办公，月底前上线
```

### 系统不应直接报价，而应回复：

```text
已识别为项目需求报价，需要先进入客户需求表三。

当前已提取：
| 字段 | 内容 |
|---|---|
| 客户公司 | XX 公司 |
| 产品线 | 无影云电脑 |
| 部署规模 | 50 人 |
| 使用场景 | 安全办公 |
| 交付时间 | 月底前 |

还需要确认：
| 序号 | 问题 |
|---:|---|
| 1 | 选择云产品资源配置 |
| 2 | 选择时长类型 |
| 3 | 是否需要硬件设备 |
| 4 | 是否需要云盘/安全管控 |
| 5 | 预算范围和付款方式 |
```

### 字段补齐后：

```text
需求表三已基本完整，完整性评分 88 分。
建议分流：方案型。
原因：50 人规模 + 安全办公 + 月底交付，需要方案说明和交付计划。

是否生成内部方案草稿和报价草稿？
```

## 八、结论

单品报价阶段的重点是“价格准确”；项目需求报价阶段的重点是“流程正确”。

下一步最优先不是直接生成客户正式报价，而是先把：

1. 客户需求表三  
2. 需求卡  
3. 缺口追问  
4. 分流派单  

这四件事落稳。

建议下一步执行：**项目需求表三 v1 + 无影项目报价 P0 字段 + 需求卡完整性评分**。
