## 二、已发现的错误与修正记录

### 错误 1：产品导入口径错误（2026-06-24）

**问题**：使用错误的 Excel 模板导入无影产品（原演示版产品被错误回灌）。

**修正**：重新清理 `product`、`product_price`、`product_spec` 表，只保留 `P-WUYING-*` 产品和相关价格配置。

**避坑**：产品导入前必须确认：
- 使用最新官方产品清单模板
- 清理表中原有的测试/演示数据
- 校验导入的数据编码是否符合规则

### 错误 2：无影产品 Excel 分类路径错误（2026-06-24）

**问题**：文件 `不可外发-产品清单模板-商业版-V20260624.xlsx` 被错误归入 `原始数据/02-产品主数据`，不符合项目分类规范。

**修正**：移动到 `原始数据/03-报价库存` 目录。

**避坑**：原始数据文件必须严格按照以下分类存放：
- `01-基础档案/`：部门、员工、客户等静态数据
- `02-产品主数据/`：产品分类、产品基础信息
- `03-报价库存/`：产品价格、库存数据、产品清单 Excel
- `04-历史数据/`：历史报价、历史项目、FAQ

### 错误 3：产品编码格式破坏（2026-06-24）

**问题**：启动脚本 `init_hubai_data.py` 中的产品编码被 `.upper()` 方法强制转换为大写，导致数据库中产品编码与报价规则不匹配（如 `P-wuying-standard` 变成 `P-WUYING-STANDARD`）。

**修正**：修改初始化脚本，确保产品编码原样保留。

**避坑**：任何数据处理脚本不得对以下字段进行大小写转换：
- `product_code`
- `spec_code`
- `customer_code`
- `policy_code`
- 任何业务编码字段

### 错误 4：旧折扣策略干扰报价（2026-06-24）

**问题**：导入产品价格时，旧的 `product_price` 数据被错误的折扣策略覆盖，导致部分产品报价为 0。

**修正**：重新导入产品价格，禁用旧的折扣策略。当前有效的报价策略为 `POL-WUYING-NO-DISCOUNT`（折扣率 1.0）。

**避坑**：产品价格导入前必须：
- 清除表中所有旧数据
- 确保导入价格为最新有效价格
- 验证价格策略的正确性

### 错误 5：长方案直接聊天输出导致 OpenClaw 截断/失败（2026-06-26）

**问题**：早期在处理复杂方案文档时，直接通过 OpenClaw 的最终回复返回长文本，导致文本被截断或任务执行失败。

**修正**：修改为"分布式小块读取 + 文件化输出"工作流程：
1. 首先进行关键词搜索/目录列表
2. 每次只读取小片段相关内容
3. 总结成要点后写入 Markdown 文件
4. 聊天中只返回文件名和简要说明

**避坑**：所有长方案、报告、整体设计类输出必须：
- 写入本地 Markdown 文件
- 文件名格式：`{name}_{version}_{timestamp}.md`
- 聊天中只返回文件路径和简要内容

### 错误 6：需求表 Schema 文档与实际数据库不一致（2026-06-24 至 2026-07-01）

**问题**：`order_requirement_table_schema_v2_202606241534.md` 文档记录主表有 64 字段、产品数 178，但实际迁移脚本和数据库显示有 90 列、产品数 177。

**修正**：以最新迁移脚本（`scripts/migrate_project_requirement_m1.py`）和实际数据库为准，废弃过时的 Schema 文档。

**避坑**：数据库 Schema 变更后必须：
- 更新对应的迁移脚本
- 更新初始化脚本
- 验证实际数据库与设计一致
- 废弃过时的文档版本

### 错误 7：钉钉前置拦截器未复用统一分类器（2026-07-01）

**问题**：早期 `dingtalk-connector` 内置的无影单品拦截器直接调用 `price_engine.query_single_item`，没有经过统一入口分类器，可能导致客户需求被错误地当作单品查询处理。

**修正**：更新 `dingtalk-connector` 代码，确保所有无影相关的报价请求都先经过 `classify_quotation_entry` 分类，再路由到相应处理逻辑。

**避坑**：任何渠道的报价入口必须：
- 统一使用 `classify_quotation_entry` 分类器
- 禁止绕过客户需求表三直接出正式报价
- 单品查询必须明确标注"内部参考"

## 三、给后续开发者的避坑建议

### 3.1 架构层面

| 建议 | 理由 | 行动点 |
|------|------|--------|
| 严格遵守"认知与执行分离"原则 | 大模型有幻觉风险，不能让其直接计算价格、折扣、成本 | 所有业务计算必须在 Skill 层实现（Python/SQL/规则引擎） |
| 统一报价入口分类 | 避免不同渠道有不同的分类逻辑 | 所有渠道必须复用同一个 `classify_quotation_entry` 函数 |
| 客户需求表三必经 | 确保正式报价信息完整性 | 任何正式报价请求必须先进入需求表三，无法跳过 |
| 历史项目仅作为参考 | 防止历史数据影响当前报价规则 | 历史案例只做风险提示，价格必须按当前规则计算 |

### 3.2 数据库层面

| 建议 | 理由 | 行动点 |
|------|------|--------|
| 使用统一的初始化/迁移脚本 | 防止 Schema 不一致 | 所有库表变更必须通过脚本实现，禁止手动修改 |
| 禁用旧产品数据回灌 | 防止旧演示数据干扰报价 | 初始化时先清除旧数据，只保留当前产品信息 |
| 保留业务编码原样 | 防止编码格式导致查询失败 | 禁止对业务编码进行大小写转换或格式化 |
| 验证价格数据完整性 | 确保无价格不报价原则落地 | 价格导入后必须验证完整性和有效性 |

### 3.3 开发流程层面

| 建议 | 理由 | 行动点 |
|------|------|--------|
| 短方案直接回复，长方案文件化输出 | 防止 OpenClaw 截断/失败 | 超过 1000 字的内容必须写入 Markdown 文件 |
| 废弃过时文档 | 防止版本混乱 | 新版本发布后，明确标记旧版本为废弃并给出替代 |
| 测试覆盖所有通道和场景 | 确保入口分类和报价输出一致 | 测试覆盖：Web/MVP/Skill/Bridge 四层 |
| 定期更新运维监控 | 确保系统稳定运行 | 检查服务状态、数据库连接、API 响应 |
