# HubAI 报价工作流 v3.0 执行计划

> 时间：2026-06-25 09:52  
> 对应方案：`hubai_quotation_workflow_design_v3_202606250952.md`  
> 核心更新：Web 端 + Agent 端统一入口；客户需求表三必需；历史项目仅作为报价参考，不作为报价规则。

---

## 一、执行目标

把 HubAI 报价工作流从 v2.0 的“订单 AI 表格入口”升级为 v3.0 的“多入口统一需求接入”。

最终实现：

1. 报价入口覆盖 Web 端和 Agent 端。
2. Agent 端通过 skill 和 MVP 接入，包含钉钉、企微等。
3. 客户需求报价单必须先完成客户需求表三。
4. 历史项目、历史报价、类似案例只作为参考，不作为报价规则。
5. 单品价格查询保留为快速报价/参考报价；客户需求报价必须走需求表三。

---

## 二、实施范围

### 2.1 本次纳入

- Web 端：销售助理/商务助理页面中的客户需求表三入口。
- Agent 端：通过 API/skill/MVP 提交需求表三。
- 渠道预留：钉钉、企微、普通 Agent、邮件、电话、导入。
- 数据层：`order_requirement`、`order_requirement_item`、后续需求卡/报价关联字段。
- 流程层：需求表三 → 人工确认 → 需求卡 → 分流 → 历史参考 → 输出确认 → 报价/方案。

### 2.2 本次不纳入

- 不新增 PostgreSQL/Qdrant 等外部数据库。
- 不把历史项目写成规则引擎条件。
- 不允许 Agent 端绕过需求表三直接生成正式客户报价。
- 不做完整钉钉/企微生产级认证上线，仅先完成 MVP 接入边界和数据模型。

## 四、任务拆解

### A. 数据库与模型

| 任务 | 内容 | 文件/对象 | 优先级 |
|---|---|---|---|
| A1 | 确认 `order_requirement` 作为客户需求表三主表 | `scripts/init_hubai_db.py` | P0 |
| A2 | 扩展渠道枚举 | `source_channel`: web/agent/dingtalk/wecom/email/phone/import | P0 |
| A3 | 增加 Agent 来源引用 | `source_ref`, `created_by`, `updated_by` | P0 |
| A4 | 保留历史项目字段但限定用途 | `has_historical_project`, `historical_project_refs` | P0 |
| A5 | 报价关联需求表三 | `quotation_header.requirement_no/card_no/historical_case_refs` | P1 |
| A6 | 新增需求卡表 | `quotation_requirement_card` | P1 |

### B. Web 端

| 任务 | 内容 | 验收 |
|---|---|---|
| B1 | 将页面“客户需求报价单”文案升级为“客户需求表三” | 用户能明确看到必填入口 |
| B2 | 客户需求模式下，先保存需求表三 | 返回 `requirement_no` |
| B3 | 没有 `requirement_no` 时阻断正式客户报价 | 显示补全提示 |
| B4 | 保存成功后显示下一步 | 人工确认、负责人、生成需求卡 |
| B5 | 单品价格查询保留 | 不影响快速报价/参考报价 |

### C. Agent 端 / skill + MVP

| 任务 | 内容 | 验收 |
|---|---|---|
| C1 | 定义 Agent 采集字段 | 客户、联系人、来源、原文、附件、期望时间、负责人、紧急、历史项目 |
| C2 | Agent 调用同一接口 | `POST /api/order-requirements` |
| C3 | 钉钉入口映射 | `source_channel=dingtalk` |
| C4 | 企微入口映射 | `source_channel=wecom` |
| C5 | Agent 输出约束 | 未确认前只输出“待确认参考/草稿” |

### D. 历史项目参考隔离

| 任务 | 内容 | 验收 |
|---|---|---|
| D1 | 页面标注历史案例为“仅供参考” | 页面和输出文案明确 |
| D2 | 禁用历史最低价/最低折扣自动约束 | 报价规则不读取历史价作为硬规则 |
| D3 | 报价规则源白名单 | 产品价格、报价政策、库存、客户等级、审批结果、人工确认 |
| D4 | 历史参考写入 `historical_case_refs` | 可追溯但不参与规则决策 |

### E. 测试验收

| 用例 | 输入 | 期望结果 |
|---|---|---|
| E1 Web 客户需求表三 | Web 提交客户、原始需求 | 生成 `requirement_no`，状态 draft/pending |
| E2 Agent 普通入口 | `source_channel=agent` | 落同一表，来源正确 |
| E3 钉钉入口 | `source_channel=dingtalk` | 落同一表，来源正确 |
| E4 企微入口 | `source_channel=wecom` | 落同一表，来源正确 |
| E5 未完成需求表三直接报价 | 点击客户需求报价 | 被阻断，提示先填写需求表三 |
| E6 有历史项目 | `has_historical_project=true` | 展示历史参考，不改变报价规则 |
| E7 标准需求 | 产品/价格/库存/政策完整 | 自动生成报价草稿 |
| E8 非标/高风险 | 定制、账期长、急单 | 分流到评审/审批 |

## 六、风险控制

1. **入口变多导致数据割裂**：统一落 `order_requirement`，通过 `source_channel/source_ref` 区分。
2. **Agent 直接越权报价**：Agent 端只允许生成待确认草稿，正式客户版必须走需求表三和审批状态。
3. **历史项目被误当规则**：代码和文案双重约束，历史项目只进 `historical_case_refs`。
4. **用户误解单品报价和客户需求报价**：页面区分“单品价格查询”和“客户需求表三”。
5. **后续钉钉/企微生产接入复杂**：MVP 先完成渠道枚举和统一接口，认证、加解密、回调后续单独迭代。
